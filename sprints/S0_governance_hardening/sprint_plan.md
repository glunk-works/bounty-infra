- **Task 2: Gated OpenTofu deploy (#9)**
  - **Description:** Replace `deploy-infra.yml`'s `push→apply -auto-approve` with two gates. **(a)** A `tofu plan` job on **PRs touching `infra/**`** — `tofu fmt -check`, `tofu validate`, `tflint`, then `tofu plan` with output posted (PR comment or uploaded artifact) so the reviewer reads the plan. **(b)** An **apply job on merge-to-main** that targets a protected GitHub **`production` Environment** (required reviewer); `tofu apply -auto-approve` runs **only after** the environment approval. Create the `production` Environment with required reviewers.
  - **Target Files:** `.github/workflows/deploy-infra.yml` (split into plan-on-PR + apply-on-merge), possibly a new `plan-infra.yml`; the GitHub `production` Environment (settings/API).
  - **Acceptance:** a PR touching `infra/**` runs fmt/validate/tflint/plan and surfaces the plan; merge to main does **not** apply until a required reviewer approves the `production` environment; no path applies infra without a visible plan + an approval. Existing deploys still succeed through the new flow (verify against a no-op plan).

- **Task 3: Non-bypassable CI (#8)**
  - **Description:** Make `ci.yml` unskippable and make the ruleset that enforces it self-auditing. Five parts:
    - **(a) Drop the `paths:` filters entirely.** `ci.yml` is currently `pull_request` on `src/**` only. Path filters and required checks **deadlock**: a PR touching only `docs/` never triggers the workflow, so a required check sits pending forever and the PR can never merge. Widening the filter list does not fix this — only removing it does. The repo is small; run CI unconditionally rather than adding a paths-filter action (a new unpinned third-party dependency, which #12 is open about).
    - **(b) Adopt the job-id-as-check-context convention.** loop-orchestrator's CI jobs carry **no `name:` override**, so the job id *is* the check context (`lint`, `test`, …). bounty-infra currently sets `name: "Lint & Test"` / `name: "Hatch Build & Validate"`, which would force required checks to be those display strings and re-opens the hazard that renaming a job silently un-requires the gate. **Drop the `name:` overrides; rename the jobs to `lint` / `test`.**
    - **(c) Add an infra job** (`tofu fmt -check`, `validate`, `tflint`). Use **`tofu init -backend=false`** for validate so it needs no AWS credentials; only T2's `plan` requires real creds. Pin `tflint` — do not introduce a new `@latest` tool while #12 is open.
    - **(d) Gate `build-image.yml` (the missing half of #8).** It triggers `push → main` on `src/**` with **no dependency on `ci.yml`**, pushes `:latest` to an ECR repo declared `image_tag_mutability = "MUTABLE"` (`infra/main.tf:67`), and the task definition runs `…:latest` (`infra/main.tf:148`). So merging any `src/**` change **silently replaces the code running in production Fargate** — no CI, no plan, no approval. That is #9's shape applied to code, and gating the infra apply (T2) while leaving this open would be a false gate. Make it depend on CI green and deploy by immutable `:${{ github.sha }}` rather than mutable `:latest`.
    - **(e) Port loop-orchestrator's `ruleset-drift.yml`.** Every required check blocks a merge *only because the ruleset says so*; if the ruleset is weakened or deleted, nothing else notices — which is exactly how its total absence went unseen here until 2026-07-21. Re-read `/rules/branches/main` on a cron + `workflow_dispatch` and fail loudly when rule types or contexts go missing. **Deliberately NOT a required check and NOT a job in `ci.yml`** (loop-orchestrator FD2): a required check is required only because the ruleset says so, so making the drift guard itself required would un-require it the instant the ruleset it watches is deleted — silently, on the exact failure it exists to catch. It gates nothing and is gated by nothing.
    - Optionally reconcile the `ci.yml` py-3.11-validate / py-3.14-package drift (the closed #16) if trivial — note the drift is still present despite `41a9f85` ("align Python runtime with CI"), so confirm whether #16 was closed prematurely.
  - **Target Files:** `.github/workflows/ci.yml` (drop paths, rename jobs, add infra job), `.github/workflows/build-image.yml` (CI dependency + sha tag), a new `.github/workflows/ruleset-drift.yml`, `infra/main.tf` (task-definition image tag), the ruleset required-checks (finalized here / with T1).
  - **Check taxonomy (BI-D3 convergence target).** loop-orchestrator requires `lint`, `format-check`, `test`, `secrets-scan`, `dependency-audit`, `sbom`, `pr-title`, `architect-review`. bounty-infra's required list should converge on that set plus its local extension (`tofu-validate`, `tofu-plan`). S0 lands `lint`, `test`, `tofu-validate`, `tofu-plan`; **`secrets-scan` / `dependency-audit` / `sbom` / `pr-title` / `architect-review` are not yet adopted here and are an OPEN sequencing decision** (see Out of scope).
  - **Acceptance:** CI runs on **every** PR regardless of path; job ids equal check contexts; `lint` + `test` + `tofu-validate` + the `tofu plan` check are **required** on `main`; **no merge can ship a container image that has not passed CI**, and the deployed image is sha-pinned, not `:latest`; `ruleset-drift.yml` passes on the real ruleset and demonstrably fails when a rule is removed (test it by dispatching against a deliberately weakened copy, not by weakening the live ruleset).

- **Task 4: `run-scan.yml` injection fix (#6) + drop unused token (#10)**
  - **Description:** Rewrite the *Trigger Scan Task* step so **no `${{ github.event.inputs.* }}` is interpolated inline into a `run:` shell block.** Pass all four inputs via an `env:` block; build the ECS `--overrides` JSON with **`jq -n --arg`**, referencing quoted `"$TARGET_DOMAIN"` etc. Add a **strict hostname regex** validation on `target_domain` before use (reject malformed input early — a structural check, distinct from S1's full RoE scope in the scanner). **Remove the unused `GITHUB_TOKEN` env var and the `issues: write` permission** (#10 — the scanner uploads to S3 and makes no GitHub API call). **Verify with a `'`-containing `target_domain`** that it is rejected/neutralized, not executed.
  - **Target Files:** `.github/workflows/run-scan.yml`.
  - **Acceptance:** dispatch inputs are consumed via `env:` + `jq --arg`, never inline `${{ }}` in `run:`; a `target_domain` containing a single quote does not break out (verified); `target_domain` is hostname-validated; `GITHUB_TOKEN`/`issues: write` are gone; the workflow still launches a scan correctly for a valid domain. **This clears the #6 precondition on loop-orchestrator #18** (its `seed`/`token` inputs then ride this safe pattern).

**Security Considerations:** S0 is entirely defensive. The three High findings (#6/#8/#9) are the "green for the wrong reason" class — an unprotected `main`, a CI that doesn't run on the delivery path, and an auto-apply with no plan. Fixing them **first** is what makes S1/S2's security fixes trustworthy (they land through a real gate). **#6 severity is anticipatory, not live** (qualified 2026-07-21): `workflow_dispatch` requires repo **write** access and there is a single collaborator, so it is not currently an external privilege-boundary crossing — a principal who can dispatch could equally commit a malicious workflow. It becomes genuinely High the moment **#18** makes the workflow machine-dispatchable from loop-orchestrator and `target_domain` stops being operator-typed. Ordering is unchanged (T4 still ships before #18) but S0 is **not** firefighting. Full pre-remediation detail is preserved in private draft advisories `GHSA-59j8-c4rc-2jf4` (#6), `GHSA-pf9q-vx7g-f8gr` (#7), `GHSA-p3hr-h7cq-xp5m` (#13), to publish as each sprint closes. **T4 has an in-house reference implementation:** loop-orchestrator's `ruleset-drift.yml` is the exact safe shape — `set -euo pipefail`, every value through `env:`, all JSON via `jq`, no inline `${{ }}` in the `run:` block. No new secret, no new external surface.

**Risks & Blockers:**
- **Admin scope needed** for the ruleset (T1) and the `production` Environment (T2). If unavailable, T1/T2 can't complete — surface early.
- **The apply-gate must not break live infra delivery.** Test the new plan/apply split against a **no-op plan** before relying on it; a mis-split could strand `main`→infra.
- **Required-checks sequencing** — never require a check that doesn't exist yet (it strands the requirement). Apply the required-status-checks list only after T2/T3's jobs have run once (finalize at S0's end).
- **#6 verification is behavioral, not just structural** — prove a quote-containing input is neutralized; don't just eyeball the `jq` rewrite.
- **Method discipline from here:** every S0 change lands via a reviewed PR (branch cut from `main`, never pushed to `main`); never merge your own PR; the human's merge is the approval.
